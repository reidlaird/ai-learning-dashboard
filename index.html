Machine Learning Fundamentals

Notes from Introduction to machine learning concepts (Microsoft Learn) — the foundation everything else builds on.

Knowledge notes

🌳 What a machine learning model is core

A model is software that encapsulates a function: it calculates an output from inputs. Defining that function from data is training; using it to predict new values is inferencing.

Training data = past observations. The observed attributes are features (x, usually a vector [x1,x2,x3,...]) and the known value you want to predict is the label (y). An algorithm fits the data to find y = f(x). Predictions from the model are written ŷ ("y-hat") because they're calculated, not observed.

Example: ice cream sales — weather measurements are x, ice creams sold is y.

🌳 Types of machine learning core

Supervised learning — training data includes known label values.

TypePredictsExample
RegressionA numeric valueIce cream sales from temperature
Binary classificationOne of two classes (true/false)Diabetes risk: yes/no
Multiclass classificationOne of several classesPenguin species: Adelie / Gentoo / Chinstrap

Unsupervised learning — no labels; the model finds structure on its own. Main type: clustering, grouping observations by feature similarity (e.g., segmenting customers).

↳ Related: Reinforcement learning — the third paradigm

🌳 Regression & its evaluation metrics

Training is iterative: split data into training and validation sets, fit a function (e.g., linear regression), predict on the held-out set, compare ŷ to actual y, then adjust and repeat.

The algorithm fits a line (function) through past observations; new x values slot into it to predict y.

MetricMeaning
MAEMean absolute error — average size of errors, all weighted equally
MSEMean squared error — squaring amplifies larger errors
RMSERoot of MSE — back in the label's own units
R²Proportion of variance the model explains (closer to 1 = better)
🌱 Overfitting, underfitting & why we hold out data core

A model can fit training data too well: overfitting means it memorizes noise and quirks instead of the underlying pattern — great training scores, poor real-world predictions. Underfitting is the opposite: the model is too simple to capture the pattern at all.

Both curves score well on training data — only the left one will predict new points well.

This is why data gets split into training and validation sets: validation data is the model's first encounter with points it never saw. The telltale sign of overfitting is training error still falling while validation error starts rising. Remedies: more data, simpler models, regularization (penalizing complexity), and early stopping.

↳ Related: Regression & its evaluation metrics

🌳 Binary classification & the confusion matrix

Algorithms (e.g., logistic regression) calculate the probability of the positive class; a threshold (typically 0.5) turns that into a predicted label.

The model learns a decision boundary (dashed); which side a new observation lands on decides its predicted class.

Predictions vs. reality form the confusion matrix: true positives (TP), true negatives (TN), false positives (FP), false negatives (FN).

MetricFormulaAnswers
Accuracy(TP+TN)/totalHow often is it right overall?
Recall (TPR)TP/(TP+FN)Of actual positives, how many were caught?
PrecisionTP/(TP+FP)Of predicted positives, how many really were?
F12×(P×R)/(P+R)Balance of precision & recall
AUCArea under ROC curveHow well it separates classes across all thresholds

Accuracy can mislead on imbalanced data — a model predicting "no diabetes" for everyone can still score high accuracy.

🌳 Multiclass classification

Predicts one of several possible classes. Two common approaches:

One-vs-Rest (OvR) — train a binary function per class ("Gentoo or not?"), pick the highest score. Multinomial — a single function (e.g., softmax) outputs a probability distribution across all classes, e.g. [0.2, 0.7, 0.1], which sums to 1.

🌳 Clustering (unsupervised)

Groups unlabeled observations by feature similarity. The most common algorithm is k-means: pick k centroids, assign each point to its nearest centroid, move centroids to their cluster's mean, repeat until stable.

No labels needed — the algorithm discovers the groups itself. ✕ marks each cluster's centroid.

Evaluation is different without labels — metrics measure cluster separation, e.g. average distance to cluster center and silhouette (−1 to 1; closer to 1 = tight, well-separated clusters).

🌳 Deep learning (the bridge to the next section)

An advanced form of ML using artificial neural networks — layers of neurons, each a function applying a weight (w) to inputs, wrapped in an activation function that decides whether the signal passes on. Many layers = a deeply nested function, hence "deep".

Watch signals flow forward: each connection carries a weight; each node sums its weighted inputs and passes them through an activation function.

Training: feed features forward through layers → compare output ŷ to known y with a loss function → use gradient descent to work out how each weight should change → backpropagate the adjustments → repeat over epochs. Data is processed as matrices, which is why GPUs matter.

Deep Learning — and how it differs from classical ML

Your stated goal: understand what makes deep learning distinct. Short version — classical ML fits relatively simple functions to features you define; deep learning stacks layers of neurons that learn the features themselves.

Knowledge notes

🌿 Classical ML vs deep learning key distinction
Classical MLDeep Learning
FeaturesHand-picked / engineered by humansLearned automatically by hidden layers
Best dataStructured / tabularUnstructured: images, audio, text
Data needsCan work with small datasetsUsually needs large datasets
ComputeModest (CPU is fine)Heavy — GPUs for matrix math
InterpretabilityOften explainable (e.g., decision trees)Largely a "black box"
ExamplesLinear/logistic regression, k-means, decision treesCNNs, RNNs, transformers

The human-labor step disappears: hidden layers discover the useful features themselves.

Deep learning is a subset of ML — same goal (y = f(x)), but f is a deeply nested network of weighted functions rather than a single fitted equation.

🌿 Anatomy of a neural network

Neuron: computes a weighted sum of inputs, passes it through an activation function (e.g., ReLU, sigmoid) that decides whether/how strongly to fire. Layers: input layer (one neuron per feature) → hidden layers → output layer. When every neuron connects to every neuron in the next layer, it's a fully connected network.

Without a non-linear activation, stacked layers would collapse into one linear equation — the "bend" is what gives depth its power.

For classification, the output layer typically uses softmax to produce a probability per class — e.g., [0.2, 0.7, 0.1] means class 1 is most probable.

🌿 How a network learns: loss, gradient descent, backprop

1. Weights start random. 2. Feed training features forward → get ŷ. 3. A loss function quantifies how far ŷ is from the true y. 4. Because the network is one big differentiable function, calculus can determine each weight's influence on the loss — gradient descent nudges each weight downhill. 5. Adjustments are backpropagated through the layers. 6. Repeat over epochs until loss is acceptably low.

Each step follows the slope downhill; steps shrink as the ground flattens near the minimum.

Mental model: a hiker descending a foggy valley by always stepping in the steepest downward direction.

🌿 Architectures to know

CNNs (convolutional) — slide learned filters across images to detect edges → shapes → objects; the workhorse of vision. RNNs/LSTMs — process sequences step by step; largely superseded for language. Transformers — use attention to weigh how every part of an input relates to every other part in parallel; power modern LLMs and increasingly vision too.

Computer Vision

Teaching machines to "see" — built almost entirely on the deep learning concepts in the previous tab.

Knowledge notes

🌱 Images are just numbers core

A grayscale image is a 2-D array of pixel intensities (0–255); a color image is three stacked arrays — red, green, blue channels. Every vision technique is math on these arrays. Classic image processing applies filters (small matrices convolved across the image) to blur, sharpen, or detect edges.

To a model, an image is nothing but this grid of numbers — three grids for color.

🌱 CNNs: learned filters

A convolutional neural network learns its filter values during training instead of using hand-designed ones. Early layers detect edges and textures; deeper layers combine them into shapes and whole objects. Pooling layers downsample between convolutions (max-pooling keeps the strongest signal per patch), shrinking computation and making detection less sensitive to exact pixel position. The final layers act as an ordinary classifier over the extracted features.

The filter slides across the image; each position produces one value in the feature map — a match score for the pattern the filter learned.

🌱 Core vision tasks

Classification — what is in this image? Object detection — what and where (bounding boxes)? Semantic segmentation — label every pixel. OCR — read text in images. Facial analysis and image generation (diffusion models) round out the family.

🌱 Vision transformers & multimodal models

Transformers treat image patches like words in a sentence and apply attention across them — now rivaling CNNs. Multimodal models (e.g., GPT-4-class, Florence) embed images and text in a shared space, enabling captioning, visual Q&A, and search by description.

NLP & Large Language Models

From classic text analysis to the transformer-based LLMs behind generative AI.

Knowledge notes

🌱 From text to numbers core

Models can't read — text must become numbers. Tokenization splits text into tokens (words or sub-words); embeddings map each token to a vector where similar meanings sit close together ("dog" near "puppy", far from "carburetor"). Classic NLP tasks: sentiment analysis, entity recognition, translation, summarization.

Every model input makes this trip: text → tokens → vectors. Everything downstream is math on the vectors.

🌱 Transformers & attention

The 2017 transformer architecture replaced sequential RNN processing with self-attention: every token weighs its relevance to every other token simultaneously. In "the animal didn't cross the street because it was tired", attention lets the model link "it" to "animal". This parallelism made training on internet-scale text feasible.

Attention scores for the token “it” — the model learns that “animal” matters most, in a single parallel step.

↳ Related: Architectures to know · Vision transformers

🌱 LLMs, prompts & agents

An LLM is a huge transformer trained to predict the next token, which turns out to require absorbing grammar, facts, and reasoning patterns. Prompts are the input that conditions its output — prompt engineering is shaping that input deliberately. Agents pair an LLM with tools and memory so it can take actions (search, run code, call APIs), not just generate text. RAG (retrieval-augmented generation) grounds answers in your own documents to reduce hallucination.

↳ Related: What makes an LLM an agent · RAG · Anatomy of a prompt

🌱 Context windows: the model's working memory

The context window is the maximum number of tokens a model can attend to at once — prompt, conversation history, retrieved documents, and its own output all share it. Anything outside the window simply doesn't exist for the model.

Practical consequences: a token is roughly ¾ of an English word; long documents must be chunked and selectively retrieved (this is half of why RAG exists); cost and latency scale with tokens processed; and in long chats, early details can fall out of scope — which is why agents maintain separate memory outside the window.

↳ Related: RAG · From text to numbers

🌱 Why models hallucinate — and what grounding does

An LLM is trained to produce plausible next tokens, not verified ones — there is no built-in fact-checker, so fluent-but-false output (hallucination) is a natural failure mode, most likely on niche facts, citations, and numbers.

Mitigations, in order of leverage: ground the answer in retrieved sources (RAG) and require citations; lower temperature for factual tasks; allow the model to say "I don't know"; and verify high-stakes outputs with a second pass or a human. Reduced, never eliminated — design workflows accordingly.

↳ Related: RAG · Temperature & sampling

Open & Local Models

Running models on your own hardware — privacy, cost control, and a great way to learn by tinkering.

Knowledge notes

🌿 Open weights vs closed models core

Closed models (GPT-4-class, Claude) are accessed via API only. Open-weight models (Llama, Mistral, Phi, Gemma) publish their weights so anyone can download, run, and fine-tune them. "Open source" is used loosely — check the license: some restrict commercial use.

SLMs (small language models, roughly 1–14B parameters, e.g. Microsoft's Phi family) trade some capability for the ability to run on a laptop — often plenty for summarization, classification, and coding help.

🌿 Running models locally

Ollama — the easiest start: ollama run phi3 downloads and serves a model with an OpenAI-compatible API on port 11434. Foundry Local — Microsoft's on-device runtime for Windows, optimized for local hardware. LM Studio — GUI alternative.

Quantization makes this possible: compressing weights from 16-bit floats to 4–8 bits shrinks memory needs dramatically with modest quality loss. Rule of thumb: a 7B model at 4-bit needs ~4–5 GB of RAM/VRAM.

Same 7B model, three precisions — quantization is why local LLMs are possible at all.

AI Agents & RAG

The defining topic of 2026 — Gartner forecasts 40% of enterprise apps will ship task-specific agents by year end, up from under 5% in 2025. This shelf covers how LLMs go from answering to acting.

Knowledge notes

🌱 What makes an LLM an agent core

An agent = an LLM wrapped in a loop with tools, memory, and a goal. Instead of one prompt → one answer, the model plans, calls tools (search, code, APIs), observes results, and iterates until the task is done.

One prompt → one answer becomes a cycle: the loop only exits when the goal is met (or the step budget runs out).

The mental shift: RAG gives a model knowledge, fine-tuning changes its behavior, agents give it agency — multi-step action across tools.

↳ Related: LLMs, prompts & agents · RAG

🌱 Tool use, function calling & MCP

Function calling: you describe available functions in the prompt; the model outputs structured JSON naming which to call with what arguments; your code executes it and returns the result. Structured outputs guarantee the reply matches a schema.

MCP (Model Context Protocol) standardizes how models connect to tools and data sources — think "USB for AI tools." A2A covers agent-to-agent communication for multi-agent orchestration.

🌱 RAG: retrieval-augmented generation core

LLMs are frozen at training time and know nothing private. RAG fixes both: at question time, retrieve relevant chunks from your own documents and paste them into the prompt so the answer is grounded in citable sources.

The pipeline: chunk documents → embed each chunk as a vector → store in a vector database → at query time embed the question, find nearest chunks (semantic search), optionally rerank, then generate with the retrieved context. Cuts hallucination and keeps knowledge current without retraining.

↳ Related: Embeddings & vector search

🌱 Embeddings & vector search

An embedding maps text (or images) to a vector of hundreds of numbers where semantic similarity = geometric closeness. "How do I reset my password?" lands near "credential recovery steps" even with zero shared words.

A vector database (e.g., Azure AI Search, Pinecone, pgvector) indexes millions of embeddings for fast nearest-neighbor lookup. Hybrid retrieval combines vector similarity with classic keyword search — usually better than either alone.

↳ Related: From text to numbers

🌱 The 2026 build sequence: prompt → RAG → fine-tune

Industry consensus for building on LLMs: start with prompting (cheapest, fastest); add RAG when you need current or private knowledge; fine-tune only when behavior must change (tone, format, latency); distill to a smaller model for scale. Wrap in an agent only when the task genuinely needs multi-step action.

Prompting & Fine-tuning

The practical craft of getting what you want from a model — from wording a request to reshaping the model itself.

Knowledge notes

🌱 Anatomy of a prompt core

System message — standing instructions that frame the model's role and rules. User message — the actual request. Few-shot examples — 2–5 worked examples in the prompt teach format and style better than descriptions do. Parameters: temperature controls randomness (low = consistent, high = creative); max tokens caps length.

Everything the model sees is one assembled prompt — role, examples, request — plus sampling parameters.

🌱 Patterns that work

Be specific about audience, format, and length. Ask for step-by-step reasoning on hard problems (chain-of-thought — spending tokens on intermediate reasoning measurably improves accuracy). Give positive and negative examples. Use delimiters or XML tags to separate instructions from data. Iterate — treat prompts like code: version them, test them, refine.

↳ Related: The 2026 build sequence

🌱 Temperature & sampling, visually

At every step the model produces a probability for each possible next token. Temperature reshapes that distribution before sampling: low temperature sharpens it (the favorite almost always wins → consistent, factual); high temperature flattens it (underdogs get real chances → varied, creative). Near 0 is effectively deterministic.

Same model, same prompt — temperature only changes how boldly it samples from its own predictions.

Rules of thumb: extraction, code, and factual Q&A → 0–0.3; general chat → ~0.7; brainstorming and fiction → 1.0+. Related knobs: top-p samples only from the smallest set of tokens whose probabilities sum to p.

🌱 Fine-tuning, LoRA & distillation

Fine-tuning continues training a model on your examples to change behavior — tone, output format, domain reasoning. Full fine-tuning updates every weight (expensive); LoRA/QLoRA train a thin adapter layer instead — the 2026 default, cheap enough to run on consumer GPUs.

Distillation: use a large model's outputs to train a small one that mimics it — how many SLMs get their capability. Rule of thumb: fine-tune for behavior, RAG for knowledge; they combine well.

↳ Related: Open weights vs closed models

Reinforcement Learning

The third paradigm alongside supervised and unsupervised learning — learning by trial, error, and reward. It beat Go champions, and (as RLHF) it shapes every chat model you use.

Knowledge notes

🌱 The RL loop core

An agent observes the state of an environment, takes an action, and receives a reward. It learns a policy — a mapping from states to actions — that maximizes cumulative reward over time. No labels: just consequences.

The whole field in one loop: act, get consequences, update the policy, repeat.

The core tension is exploration vs exploitation — try new actions to discover better strategies, or repeat what already works? Compare: supervised learning is "here's the right answer"; RL is "here's a score, figure out why."

↳ Related: Types of machine learning

🌱 Key ideas & algorithms

Value-based methods (Q-learning) learn how good each state-action pair is, then act greedily. Policy-gradient methods (PPO) adjust the policy directly. Deep RL uses neural networks to handle huge state spaces — DQN played Atari from raw pixels (2013); AlphaGo combined deep RL with tree search to beat Lee Sedol (2016).

The reward function is the hard part: agents optimize exactly what you measure, not what you meant — "reward hacking" is the canonical failure mode.

🌱 RLHF: why RL matters for LLMs

Reinforcement Learning from Human Feedback is the finishing step that turned raw next-token predictors into helpful assistants. Humans rank pairs of model outputs → a reward model learns those preferences → RL (typically PPO) tunes the LLM to maximize that learned reward. It's why ChatGPT (2022) felt different from GPT-3 (2020).

↳ Related: Fine-tuning, LoRA & distillation

AI History

Seventy-five years in one shelf — worth knowing because today's debates (hype vs winter, symbolic vs neural) have all happened before.

1950
Alan Turing's "Computing Machinery and Intelligence" proposes the imitation game — can a machine's conversation pass as human?
1956
Dartmouth workshop: John McCarthy coins the term "artificial intelligence." The field is born optimistic.
1958
Rosenblatt's perceptron — the first trainable artificial neuron, ancestor of every network in this library.
1969–70s
Minsky & Papert prove single-layer perceptrons can't solve simple problems (XOR). Funding collapses — the first AI winter.
1986
Backpropagation popularized by Rumelhart, Hinton & Williams — multi-layer networks become trainable, answering the 1969 critique.
1997
IBM's Deep Blue defeats world chess champion Garry Kasparov — search and compute beat intuition, at least at chess.
2012
AlexNet crushes the ImageNet competition using a CNN on GPUs — the deep learning era begins overnight.
2016
DeepMind's AlphaGo beats Lee Sedol at Go using deep reinforcement learning — a decade ahead of predictions.
2017
"Attention Is All You Need" introduces the transformer — the architecture behind every modern LLM.
2018–20
BERT and GPT scale transformers up; GPT-3 (2020) shows that scale alone unlocks few-shot learning.
2022
ChatGPT (RLHF-tuned GPT) and diffusion image models take generative AI mainstream — fastest-growing consumer product in history.
2023–24
GPT-4-class multimodal models; open-weight Llama family democratizes access; models start seeing, hearing, and speaking.
2025–26
Reasoning models, agents with tool use (MCP), and capable on-device SLMs — AI shifts from answering to acting.
